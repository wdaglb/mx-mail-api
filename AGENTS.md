# mx-mail-api 协作说明

## 项目定位

`mx-mail-api` 是一个以后端 Go + Gin、前端 React + Ant Design 6.x 为主要技术栈的 Mx 邮件协议服务。

本项目的核心目标是围绕 Mx 邮件协议提供稳定、可测试、可扩展的服务端能力，包括但不限于协议请求接入、参数校验、业务处理、错误响应和后续协议能力扩展。

## 实现原则

- 以 Mx 邮件协议语义为准，新增接口、处理器或数据结构前必须先明确对应的协议用途和边界。
- 协议相关逻辑应优先保持可读、可查、可测，避免为了通用化过早抽象。
- 后端默认使用 Gin 组织 HTTP 路由、中间件和协议接口；前端默认使用 React + Ant Design 6.x 实现管理界面和交互。
- 前后端接口字段、错误码和状态语义应保持一致；当协议响应结构变化时，需要同步评估前端页面、类型定义和测试影响。
- 开放接口（`/openapi`）的路径、认证方式、请求字段、响应字段、错误码或行为语义发生变化时，必须同步更新根目录 `API.md`；前端公开文档页应通过 Gin 文档接口读取该文件，禁止再维护 `web/public/API.md` 这类副本文档，避免公开文档与实际接口漂移。若开放接口依赖后台数据状态（如域名是否禁用、用户是否允许永久邮箱、可选租赁时间），这些状态规则变化也必须同步写入 `API.md` 的行为说明。
- 域名新增必须先完成 TXT 所有权验证；TXT 记录名使用随机字母数字前缀并落在待验证域名之下，TXT 记录值按当前用户 ID、用户名与 YAML 中现有 smtp 配置拼接后计算 MD5 生成，不增加业务前缀。TXT 验证查询必须以用户填写的待验证域名为 DNS 边界；例如用户填写 `edss.bbroot.com` 时，只能查询该子域自身配置的 NS 和 TXT，不能自动退到 `bbroot.com` 或更上级域名。修改域名新增流程、可用域名判断、临时邮箱租赁前置条件时，需要同步检查开放接口文档中的域名前置说明。
- 前端用户可见提示语、按钮、表单标签、空状态、错误提示和说明文案必须面向普通用户表达，优先说明“用户要做什么 / 会发生什么 / 如何解决”，不要把 `QoS`、`RPM`、`Token`、`JWT`、`SMTP`、`RCPT`、`payload`、调用链路、后端查询等实现细节直接暴露给用户；代码注释、类型字段和开发文档可保留必要技术术语。
- 对协议兼容、历史行为、特殊错误码或第三方约束等容易误解的逻辑，必须在代码注释中说明原因和边界。
- 健康检查、启动配置、路由注册等基础能力应保持无业务副作用，方便本地测试、容器探针和部署平台接入。
- Docker Compose 部署配置必须同时覆盖 HTTP 管理端、SMTP 收件端和 Postgres；调整端口、数据库 DSN、配置挂载或前端静态资源服务方式时，需要同步检查 `docker-compose.yaml`、`Dockerfile` 和 `docker/config.yaml`。
- 新增公共函数、关键处理器、协议解析函数时，应补充规范注释，说明用途、参数、返回值和失败条件。

## 代码结构约定

- `main.go` 仅保留服务启动、路由装配等入口逻辑；当协议处理逻辑变多时，应拆分到独立包。
- 路由处理器应避免直接堆叠复杂业务流程；协议解析、校验和业务执行应按职责拆分。
- 前端代码应与后端服务代码按目录隔离，避免把构建产物、页面逻辑和 Go 服务入口混在同一层级。
- Ant Design 相关实现应遵循 6.x API；遇到废弃属性或组件迁移时，应优先按 6.x 推荐写法调整，而不是增加兼容性绕路。
- 单文件接近 300 行时，应优先评估拆分；超过 500 行仍不拆分时，需要在交付说明中说明原因。

## 验证要求

- 修改服务启动、路由、协议处理、错误响应时，至少运行：

```bash
go test ./...
```

- 新增协议能力时，应同步增加针对成功路径、参数错误路径和关键边界条件的测试。
- 若某项改动只做了静态检查或无法完整验证，交付时必须明确说明未验证范围。

## 输出与提交要求

- 默认使用中文沟通，结论先行，明确说明改动内容、验证结果和未完成风险。
- 需要提交代码时，commit message 必须直接概括实际改动，不得加入 AI 生成来源、署名或协作者尾巴。
